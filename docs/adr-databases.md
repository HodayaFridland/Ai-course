# ADRs — בחירת מסד נתונים לכל שירות (Polyglot Persistence)

מסמך זה מרכז את ה-Architecture Decision Records עבור בחירת משפחת מסד הנתונים לכל שירות, בשפה של שיעור מסדי הנתונים: **ACID, BASE, CAP, consistency model**.

**תזכורת מושגים קצרה:**
- **ACID** (רלציוני): Atomicity, Consistency, Isolation, Durability — טרנזקציות "הכול או כלום", consistency חזק. מתאים לכסף/מלאי.
- **BASE** (NoSQL רבים): Basically Available, Soft state, Eventually consistent — זמינות וגמישות על חשבון consistency מיידי.
- **CAP**: בזמן פיצול רשת (Partition) אפשר לבחור **C** (consistency) או **A** (availability), לא את שניהם. מכאן "מערכת CP" מול "מערכת AP".

טבלת סיכום:

| שירות | משפחה | טכנולוגיה | מודל עקביות | עמדת CAP |
|---|---|---|---|---|
| ProductCatalogService | Document | MongoDB | Eventual (BASE) | AP |
| OrderService | Relational | SQL Server | Strong (ACID) | CP |
| InventoryService | Relational | SQL Server | Strong (ACID) | CP |
| NotificationService | Document | MongoDB | Eventual (BASE) | AP |
| (מטמון) Cache | Key-Value | Redis | — (מטמון, לא מקור אמת) | — |

---

## ADR-001 — ProductCatalogService → MongoDB (Document)

**הקשר.** קטלוג מוצרים שבו לכל קטגוריה תכונות שונות: לחולצה יש מידה/צבע/בד, לספר יש מחבר/מספר עמודים/שפה, למוצר אלקטרוני יש אחריות/סוללה. הקריאות רבות (דפדוף), הכתיבות מעטות יחסית.

**אפשרויות.**
1. רלציוני (SQL) — טבלת מוצרים רחבה עם המון עמודות NULL, או EAV/טבלאות תכונות עם JOINים מרובים.
2. Document (MongoDB) — כל מוצר הוא מסמך עם שדות `Attributes` גמישים.

**החלטה.** MongoDB.

**הצדקה.** מודל המסמך תואם ישירות את הצורך: כל מסמך נושא רק את התכונות שלו (`{"Size":"L","Color":"Blue"}`) בלי סכימה קשיחה ובלי עמודות ריקות. אין JOINים — מוצר שלם נקרא במסמך אחד, מהיר לדפדוף. ה-consistency הנדרש חלש: אם עדכון מחיר נראה חלקיק שנייה מאוחר לחלק מהקוראים — לא נורא (**BASE**, eventual consistency). בעמדת **CAP** נעדיף **A** (זמינות לדפדוף) — מכאן **AP**. זהו גם ה"מוצר הכי חם" בעומס קריאה, ולכן נכניס לו מטמון (ראו Redis) ו-2 רפליקות בשלב 3.

**מחיר/סיכון.** אין טרנזקציות חוצות-מסמכים חזקות כמו ב-SQL — אבל לקטלוג לא צריך אותן.

## ADR-002 — OrderService → SQL Server (Relational, ACID)

**הקשר.** הזמנות מייצגות כסף: סכום כולל, שורות הזמנה, מחיר ליחידה. הזמנה עם כמה שורות חייבת להישמר במלואה או בכלל לא.

**אפשרויות.**
1. Document (Mongo) — הזמנה כמסמך יחיד; קל, אבל consistency חלש בין הזמנות/דוחות.
2. רלציוני (SQL) — טבלאות `Orders` + `OrderItems`, טרנזקציה אחת.

**החלטה.** SQL Server.

**הצדקה.** כסף דורש **ACID**: ה-Atomicity מבטיח שההזמנה + כל שורותיה נכתבות יחד בטרנזקציה אחת; ה-Consistency שומר על שלמות (סכום = סכום השורות); דרושה עקביות חזקה (**strong consistency**) כי אסור שדוח יראה חצי הזמנה. בעמדת **CAP** נעדיף **C** על פני **A** — עדיף לדחות מאשר לרשום כסף שגוי — כלומר מערכת **CP**. בונוס: זה גם ממחזר את הידע הקיים ב-EF Core מהמונוליט.

**מחיר/סיכון.** סכימה קשיחה יותר, אבל למבנה הזמנה יציב זה יתרון ולא חיסרון.

## ADR-003 — InventoryService → SQL Server (Relational, ACID) — DB נפרד

**הקשר.** ניהול מלאי: כמה יחידות זמינות, כמה שמורות. שני לקוחות שמנסים לתפוס את הפריט האחרון בו-זמנית — אסור ששניהם יצליחו (מכירת יתר / overselling).

**אפשרויות.**
1. Redis (key-value, `DECR` אטומי) — מהיר מאוד, אטומי לפעולת מונה בודדת.
2. רלציוני (SQL) — טרנזקציה עם עדכון מותנה (`UPDATE ... WHERE AvailableStock >= qty`).

**החלטה.** SQL Server, במסד נתונים **נפרד** מההזמנות (`inventorydb`).

**הצדקה.** שמירת מלאי דורשת **אטומיות** אמיתית כדי למנוע מכירת יתר. מימשנו זאת ב-UPDATE מותנה בודד עטוף בטרנזקציה, כך שהשמירה היא "הכול או כלום" על פני כל פריטי ההזמנה (ראו `StockService.ReserveAsync`). זו עקביות חזקה (**CP**), בדיוק מהסיבה של ההזמנות. חשוב: אף על פי שגם Order וגם Inventory הם SQL, הם **מסדי נתונים נפרדים** — אף שירות לא נוגע ב-DB של השני (עקרון database-per-service). *(Redis היה חלופה לגיטימית ל-`DECR` אטומי; בחרנו SQL כי הוא נותן גם טרנזקציה חוצת-פריטים וגם עקביות עם שפת ה-ACID של ההזמנות.)*

**מחיר/סיכון.** איטי יותר מ-Redis לפעולת מונה בודדת — לא מהותי בקנה המידה של הפרויקט.

## ADR-004 — NotificationService → MongoDB (Document, BASE)

**הקשר.** לוג התראות שנשלחו ללקוח (על איזו הזמנה, סטטוס, מתי). בעיקר כתיבות append-only וקריאה של היסטוריה. אין קשרים מורכבים.

**אפשרויות.**
1. רלציוני (SQL) — אפשרי, אבל סכימה קשיחה ללוג פשוט זה overkill.
2. Document (Mongo) — כל התראה מסמך עצמאי.

**החלטה.** MongoDB.

**הצדקה.** לוג התראות הוא append-only ללא יחסים — מסמך עצמאי מתאים בול. עקביות חזקה מיותרת: אם התראה נשמרת/נקראת חלקיק שנייה מאוחר — אין נזק (**BASE**, eventual consistency). בעמדת **CAP** נעדיף **A** — עדיף שההתראה תישמר ותהיה זמינה מאשר שהמערכת תיחסם — כלומר **AP**. זו גם ההחלטה שממלאת את דרישת ה"NoSQL נוסף לבחירתך" מעבר לקטלוג.

**מחיר/סיכון.** אין טרנזקציות חזקות — לא נדרשות לכתיבת לוג.

## הערה — Redis (Key-Value) מגיע "בחינם"

בשלב 4 נוסיף מטמון **Redis** (משפחת key-value) לקריאות הקטלוג בדפוס cache-aside. גם אם לא היינו מתכננים זאת במיוחד, זה כבר נותן לנו **משפחת NoSQL שלישית** במערכת. Redis כאן אינו מקור האמת — רק שכבת מטמון מעל Mongo — ולכן אין לו "עמדת CAP" משלו לצורך הדיון.

**סיכום Polyglot:** שלוש משפחות מסדי נתונים בשימוש — **רלציוני** (Order, Inventory), **מסמכים** (Catalog, Notification), ו-**key-value** (Redis) — כל אחת נבחרה לפי צורכי ה-consistency והגישה של השירות.
